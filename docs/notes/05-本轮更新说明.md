# 本轮更新说明

## 1. 本轮补充的重点

本轮在原有基础骨架上，继续把“接口存在”推进到“服务实现可落地”的程度，重点补充了：

- 通用响应层补齐 `to_ajax`、`warn`
- 业务模块真正接入数据库会话
- 顾客、搭配、穿搭、主数据、分类树、OSS 的基础 CRUD / 状态更新逻辑
- 系统管理三件套：
  - `/system/user/*`
  - `/system/role/*`
  - `/system/menu/*`
- 个人资料、修改密码、头像上传
- 用户 / 角色 Excel 导出
- 兼容无 SQLAlchemy 依赖时的导入降级，保证文档与路由测试可跑通

## 2. 本轮已经处理过的问题

### 2.1 `/common/uploads` 契约

已按 Java 现状保持为 **POST** 多文件上传。

### 2.2 统一响应缺口

原始骨架里部分服务已经在调用 `to_ajax`、`warn`，但公共响应层没有提供实现，导致应用启动阶段就可能 import 失败；本轮已补齐。

### 2.3 SQLAlchemy 依赖导入问题

考虑到某些开发环境可能还没执行 `uv sync`，本轮把模型和部分服务的导入方式改成了“优先真实依赖、否则走兼容层”，这样至少能先跑文档测试与路由测试。

## 3. 当前仍建议优先做的事

1. 在本机执行 `uv sync`
2. 连真实 MySQL 5.7 跑 `scripts/inspect_db.py`
3. 对照真实表结构核对：
   - `t_customer`
   - `t_matching`
   - `ai_join / ai_outfit / ai_task`
   - `md_product / md_product_color / md_tag`
   - `clothing_category`
   - `oss_file`
   - `sys_user / sys_role / sys_menu / sys_user_role / sys_role_menu`
4. 用真实库对每个模块补契约快照测试
5. 再决定 Alembic baseline 与新库迁移脚本
